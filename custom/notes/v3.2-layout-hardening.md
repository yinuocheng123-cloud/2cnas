# 版本记录：v3.2-layout-hardening

## 1. 本轮目标

本轮目标不是新增业务功能，而是提升 CNAS 成交页在低分辨率、Windows 缩放、旧桌面设备和窄宽度环境下的布局稳定性。

重点解决部分老台式电脑或浏览器缩放后，桌面页面被挤压得像手机布局、卡片压缩、二维码区域横向拥挤、CTA 和 footer 错位的问题。

本轮属于上线后的布局加固，不改变路径判断、企业微信承接、在线表单、A/B/C 诊断、维护模块和 FAQ。

## 2. 背景与上下文

本轮承接 `v3.1-wecom-and-maintenance-upgrade.md` 与 `post-launch-operation-check.md`：

- v3.1 已完成企业微信优先承接、在线表单兜底、A/B/C 诊断、CNAS认可后维护模块和 FAQ 增强。
- 上线后记录显示成交页已接入主站 `/api/lead`，企业微信二维码已替换，备案号为 `浙ICP备2020044218号-3`。
- 当前页面为纯静态 HTML/CSS/JS 项目，不需要新增依赖或运行开发服务。

本轮不修改 `script.js`，避免影响表单提交、诊断结果生成和 payload 结构。

## 3. 问题分析

修改前的主要风险来自三类布局条件：

1. 断点不够清晰：原先以 `980px` 和 `640px` 作为主要断点，无法准确覆盖 `769~1024` 平板区间，也容易让旧桌面设备或 125% 缩放后的窗口处于尴尬宽度。
2. 多处 grid 使用固定列数：例如三列、四列模块在窄桌面和缩放环境下容易压缩卡片内容。
3. 二维码和底部承接区存在横向强挤压风险：底部二维码卡片、handoff 二维码卡片和 footer 三栏在窄宽度下需要更保守地换行或纵向排列。

核心矛盾不是业务内容过多，而是布局规则在桌面、平板、手机之间缺少稳定过渡。

## 4. 候选方案比较

### 方案一：只调整媒体查询断点

核心思路：把 `980px/640px` 改为 `1024px/768px`。

优点：改动最小。

缺点：固定列 grid 仍可能在 1366 缩放、窄桌面和旧浏览器中被挤压。

结论：仅作为基础，不足以完成本轮目标。

### 方案二：重构页面 HTML 结构

核心思路：重新拆分模块，给桌面、平板、手机分别写不同结构。

优点：可控性强。

缺点：风险过高，容易影响业务主线、表单、诊断和已上线结构；也不符合“不重构页面”的要求。

结论：放弃。

### 方案三：CSS 层布局加固

核心思路：保留 HTML 和 JS，只在 CSS 中重整断点、grid、二维码卡片、footer 和溢出控制。

优点：改动范围小，业务逻辑稳定；能覆盖低分辨率和缩放环境；符合本轮上线后收口定位。

缺点：无法替代真实老设备上的人工视觉检查。

结论：采用。

## 5. 最终决策

最终采用 CSS 层加固方案：

- 明确断点：`<=768px` 为手机，`769~1024px` 为平板，`>1024px` 为桌面。
- 把多处固定列 grid 改为 `repeat(auto-fit, minmax(...))`，避免卡片被固定四列强挤。
- 为关键 grid 子项补充 `min-width: 0`，避免内容撑破容器。
- 对页面整体设置横向溢出保护，降低旧桌面设备出现横向滚动的风险。
- 二维码区域在平板和手机端优先纵向或居中展示，避免被文字挤压。

## 6. 具体实现

涉及文件：

- `styles.css`
- `custom/notes/v3.2-layout-hardening.md`

具体改动：

- `html` 与 `body` 增加横向溢出保护。
- 文本段落增加 `overflow-wrap`，降低长文本或特殊字符撑破容器的风险。
- 导航、按钮、品牌区增加收缩保护，避免旧桌面宽度下 CTA 或导航文字错位。
- 首屏、问卷区、FAQ、handoff、footer 等关键 grid 子项统一增加 `min-width: 0`。
- 将多个固定列 grid 改为 `auto-fit + minmax`，包括表单字段、选项、结果卡片、系统卡片、指标、场景、流程、维护模块和知识卡片。
- 将原 `980px` 响应式断点调整为 `1024px`，作为平板与窄桌面保护区。
- 将原 `640px` 手机断点调整为 `768px`。
- 在平板区间隐藏导航链接，避免 1024 附近被挤压。
- 在平板区间让主问卷布局、FAQ、handoff 等从双列切换为单列。
- 在手机区间让底部二维码卡片改为纵向居中展示，提升扫码稳定性。
- footer 在平板区间调整为更稳的两栏加服务内容换行，在手机区间改为单列居中。

## 7. 本轮优点

- 未改动业务逻辑，表单和诊断流程风险低。
- 布局断点更符合本轮要求，平板和手机边界更清晰。
- 固定四列被替换为自适应 grid，低分辨率和 125% 缩放下更不容易压缩卡片。
- 二维码区域更稳定，手机端和窄宽度下不再与文字横向强挤。
- footer 在窄宽度下更像完整转化区，不容易出现错位或空洞。

## 8. 本轮缺点与代价

- 本轮没有重做页面视觉，因此只解决布局稳定性，不解决整体视觉风格的进一步升级。
- `auto-fit` 会让部分模块在中间宽度下自动变为两列或三列，视觉节奏可能与原固定列略有差异。
- 浏览器自动化环境无法替代真实 Windows 125% 缩放、旧 Chrome 和 Edge Chromium 实机检查，仍建议用户在真实设备上复核。
- 为避免改动过大，本轮没有新增桌面/移动两套独立结构。

## 9. 验证与结果

已执行：

- `git pull --ff-only`
- `node --check script.js`
- 风险词扫描：`CNAS认证`、`包过`、`保证通过`、`官方指定`、`唯一`、`最权威`、`必过`、`100%通过`
- CSS 固定列扫描，确认已无 `repeat(2|3|4, ...)` 固定列写法。
- 本机静态 HTTP 方式进行浏览器视口检查。

视口检查结果：

- `1366×768`：无横向溢出，footer、二维码、备案号可见。
- `1093×614`：模拟 1366 宽度下 125% 缩放后的窄桌面，无横向溢出。
- `1024×768`：进入平板保护布局，无横向溢出，问卷区单列展示。
- `768×900`：手机边界布局正常，无横向溢出。
- `390×844`：手机布局正常，二维码和备案号可见。

说明：浏览器插件不允许直接打开 `file://` 本地文件，因此验证时使用同一目录的临时本机静态 HTTP 服务读取同一份文件；验证结束后已关闭临时服务。

## 10. 本轮结论

本轮已完成 v3.2 布局加固目标。成交页仍保持原有路径判断、企业微信承接、表单兜底、A/B/C 诊断、维护模块和 FAQ，不新增业务功能。

当前布局在 1366、模拟 125% 缩放、1024 平板、768 边界和 390 手机视口下均未发现横向溢出。

## 11. 下一轮建议

1. 用真实 Windows 台式机在 Chrome 与 Edge Chromium 中复核 125% 系统缩放和浏览器 125% 缩放。
2. 如果仍有个别老设备显示异常，优先记录具体分辨率、浏览器版本、缩放比例和截图，再做定点 CSS 修复。
3. 后续如继续优化视觉，应单独开版本处理，不与本轮布局稳定性修复混在一起。
